## 三、详细表结构设计
### 3.1 user_info（用户信息表）
#### 表说明
存储用户的基本信息，包括用户名、密码、年龄、电话等。所有用户ID全局唯一。

#### 字段设计
| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 约束 | 说明 |
|--------|------|------|----------|--------|------|------|
| `user_id` | INT | - | NO | - | PRIMARY KEY, AUTO_INCREMENT | 用户ID，全局唯一，1-100范围内随机生成 |
| `user_name` | VARCHAR | 100 | NO | - | UNIQUE KEY, NOT NULL | 用户名，唯一 |
| `user_age` | INT | - | YES | 0 | - | 用户年龄 |
| `user_phone` | VARCHAR | 20 | YES | '' | - | 用户手机号 |
| `user_pwd` | VARCHAR | 100 | NO | - | NOT NULL | 用户密码，明文存储 |
| `created_at` | DATETIME | - | NO | CURRENT_TIMESTAMP | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | - | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | NOT NULL | 更新时间 |

#### 索引设计
| 索引名 | 字段 | 索引类型 | 说明 |
|--------|------|----------|------|
| `PRIMARY` | `user_id` | 主键索引 | 用户ID，全局唯一 |
| `uk_user_name` | `user_name` | 唯一索引 | 用户名唯一 |
| `idx_user_phone` | `user_phone` | 普通索引 | 手机号登录查询 |

#### 建表SQL
```sql
CREATE TABLE `user_info` (
  `user_id` INT NOT NULL AUTO_INCREMENT COMMENT '用户ID，全局唯一',
  `user_name` VARCHAR(100) NOT NULL COMMENT '用户名',
  `user_age` INT DEFAULT 0 COMMENT '用户年龄',
  `user_phone` VARCHAR(20) DEFAULT '' COMMENT '用户手机号',
  `user_pwd` VARCHAR(100) NOT NULL COMMENT '用户密码',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`user_id`),
  UNIQUE KEY `uk_user_name` (`user_name`),
  KEY `idx_user_phone` (`user_phone`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户信息表';
```

### 3.2 chat_session（聊天会话表）
#### 表说明
存储用户的聊天会话信息，每个会话对应一个独立的聊天上下文。所有会话ID全局唯一。

#### 字段设计
| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 约束 | 说明 |
|--------|------|------|----------|--------|------|------|
| `session_id` | BIGINT | - | NO | - | PRIMARY KEY, AUTO_INCREMENT | 会话ID，全局唯一 |
| `user_id` | INT | - | NO | - | NOT NULL, FOREIGN KEY | 用户ID，关联user_info表 |
| `title` | VARCHAR | 200 | YES | '新会话' | - | 会话标题，默认第一条消息 |
| `last_message_time` | DATETIME | - | NO | CURRENT_TIMESTAMP | NOT NULL | 最后一条消息时间 |
| `created_at` | DATETIME | - | NO | CURRENT_TIMESTAMP | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | - | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | NOT NULL | 更新时间 |

#### 索引设计
| 索引名 | 字段 | 索引类型 | 说明 |
|--------|------|----------|------|
| `PRIMARY` | `session_id` | 主键索引 | 会话ID，全局唯一 |
| `idx_user_id` | `user_id` | 普通索引 | 用户会话查询 |
| `idx_user_last_time` | `user_id`, `last_message_time` | 联合索引 | 用户会话按时间排序查询 |

#### 建表SQL
```sql
CREATE TABLE `chat_session` (
  `session_id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '会话ID，全局唯一',
  `user_id` INT NOT NULL COMMENT '用户ID',
  `title` VARCHAR(200) DEFAULT '新会话' COMMENT '会话标题',
  `last_message_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '最后一条消息时间',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`session_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_user_last_time` (`user_id`, `last_message_time`),
  CONSTRAINT `fk_session_user` FOREIGN KEY (`user_id`) REFERENCES `user_info` (`user_id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='聊天会话表';
```

### 3.3 chat_message（聊天消息表）
#### 表说明
存储用户和AI的聊天消息，包括消息内容、角色、思考过程、ECharts配置等。所有消息ID全局唯一。

#### 字段设计
| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 约束 | 说明 |
|--------|------|------|----------|--------|------|------|
| `message_id` | BIGINT | - | NO | - | PRIMARY KEY, AUTO_INCREMENT | 消息ID，全局唯一 |
| `session_id` | BIGINT | - | NO | - | NOT NULL, FOREIGN KEY | 会话ID，关联chat_session表 |
| `user_id` | INT | - | NO | - | NOT NULL, FOREIGN KEY | 用户ID，关联user_info表 |
| `role` | VARCHAR | 20 | NO | - | NOT NULL | 角色：user（用户）/assistant（AI） |
| `content` | TEXT | - | NO | - | NOT NULL | 消息内容 |
| `thinking_process` | TEXT | - | YES | NULL | - | 思考过程，开启深度思考时填充 |
| `echarts_config` | JSON | - | YES | NULL | - | ECharts配置JSON数组，每个元素是一个完整的ECharts 6.0配置对象 |
| `created_at` | DATETIME | - | NO | CURRENT_TIMESTAMP | NOT NULL | 创建时间 |

#### 索引设计
| 索引名 | 字段 | 索引类型 | 说明 |
|--------|------|----------|------|
| `PRIMARY` | `message_id` | 主键索引 | 消息ID，全局唯一 |
| `idx_session_id` | `session_id` | 普通索引 | 会话消息查询 |
| `idx_user_id` | `user_id` | 普通索引 | 用户消息查询 |
| `idx_session_created` | `session_id`, `created_at` | 联合索引 | 会话消息按时间排序查询 |

#### 建表SQL
```sql
CREATE TABLE `chat_message` (
  `message_id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '消息ID，全局唯一',
  `session_id` BIGINT NOT NULL COMMENT '会话ID',
  `user_id` INT NOT NULL COMMENT '用户ID',
  `role` VARCHAR(20) NOT NULL COMMENT '角色：user/assistant',
  `content` TEXT NOT NULL COMMENT '消息内容',
  `thinking_process` TEXT DEFAULT NULL COMMENT '思考过程',
  `echarts_config` JSON DEFAULT NULL COMMENT 'ECharts配置JSON数组',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`message_id`),
  KEY `idx_session_id` (`session_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_session_created` (`session_id`, `created_at`),
  CONSTRAINT `fk_message_session` FOREIGN KEY (`session_id`) REFERENCES `chat_session` (`session_id`) ON DELETE CASCADE,
  CONSTRAINT `fk_message_user` FOREIGN KEY (`user_id`) REFERENCES `user_info` (`user_id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='聊天消息表';
```

**前端渲染逻辑**：
```javascript
// 1. 从数据库读取JSON字符串
const echartsConfig = JSON.parse(message.echarts_config);

// 2. 遍历数组，每个元素渲染一个图表
echartsConfig.forEach((config, index) => {
  const chartDom = document.getElementById(`chart-${index}`);
  const myChart = echarts.init(chartDom);
  myChart.setOption(config);
});
```

**存储规则**：
- 仅AI消息（role='assistant'）可能有值
- 用户消息（role='user'）该字段为NULL
- 每条AI消息最多存储3个图表配置
- 配置必须符合ECharts 6.0官方规范，不能包含已废弃的属性

### 3.4 mcp_jobs_cache（MCP职位数据缓存表）
#### 表说明
缓存从MCP-Jobs获取的职位数据，减少第三方API调用，提高响应速度。所有缓存ID全局唯一。

#### 字段设计
| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 约束 | 说明 |
|--------|------|------|----------|--------|------|------|
| `cache_id` | BIGINT | - | NO | - | PRIMARY KEY, AUTO_INCREMENT | 缓存ID，全局唯一 |
| `keyword` | VARCHAR | 100 | NO | - | NOT NULL | 岗位关键词 |
| `city` | VARCHAR | 50 | NO | - | NOT NULL | 城市 |
| `jobs_data` | JSON | - | NO | - | NOT NULL | 职位数据JSON |
| `created_at` | DATETIME | - | NO | CURRENT_TIMESTAMP | NOT NULL | 创建时间 |
| `expire_at` | DATETIME | - | NO | - | NOT NULL | 过期时间 |

#### 索引设计
| 索引名 | 字段 | 索引类型 | 说明 |
|--------|------|----------|------|
| `PRIMARY` | `cache_id` | 主键索引 | 缓存ID，全局唯一 |
| `uk_keyword_city` | `keyword`, `city` | 唯一索引 | 关键词+城市唯一，防止重复缓存 |
| `idx_expire_at` | `expire_at` | 普通索引 | 过期数据清理 |

#### 建表SQL
```sql
CREATE TABLE `mcp_jobs_cache` (
  `cache_id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '缓存ID，全局唯一',
  `keyword` VARCHAR(100) NOT NULL COMMENT '岗位关键词',
  `city` VARCHAR(50) NOT NULL COMMENT '城市',
  `jobs_data` JSON NOT NULL COMMENT '职位数据JSON',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `expire_at` DATETIME NOT NULL COMMENT '过期时间',
  PRIMARY KEY (`cache_id`),
  UNIQUE KEY `uk_keyword_city` (`keyword`, `city`),
  KEY `idx_expire_at` (`expire_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='MCP职位数据缓存表';
```

## 五、数据库选型理由
### 5.1 选型结论
选择 **MySQL 8.0.39** 作为项目的数据库。

### 5.2 选型理由
| 维度 | 说明 |
|------|------|
| **技术栈复用** | 现有Face Pro框架已使用MySQL，无需学习新技术，开发效率高 |
| **成熟稳定** | MySQL是最成熟的开源关系型数据库之一，社区活跃，文档丰富 |
| **事务支持** | InnoDB引擎支持ACID事务，保证数据一致性 |
| **索引优化** | 支持多种索引类型（主键、唯一、普通、联合），查询性能优秀 |
| **JSON支持** | MySQL 5.7+原生支持JSON类型，适合存储ECharts配置和职位数据 |
| **扩展性** | 支持主从复制、读写分离，未来用户量增长时可轻松扩展 |
| **运维成本** | 部署简单，运维成本低，有大量成熟的运维工具和最佳实践 |

### 5.3 为什么不选其他数据库
| 数据库 | 不选理由 |
|--------|----------|
| PostgreSQL | 功能强大，但现有框架已使用MySQL，切换成本高 |
| MongoDB | NoSQL数据库，不适合这种关系型数据结构，事务支持较弱 |
| SQLite | 仅适合开发测试，不适合生产环境，并发性能差 |

## 六、生产级注意事项
### 6.1 必须遵守的规范
1. **主键要求**：所有表必须有主键，采用`实体名_id`的命名方式（如`user_id`、`session_id`），全局唯一，优先使用自增ID
2. **时间字段**：所有业务表必须有`created_at`和`updated_at`字段，`created_at`默认当前时间，`updated_at`自动更新
3. **字符集**：统一使用`utf8mb4`字符集，支持emoji等特殊字符
4. **排序规则**：统一使用`utf8mb4_unicode_ci`排序规则，大小写不敏感
5. **外键约束**：建立外键约束保证数据一致性，级联删除避免孤儿数据
6. **索引设计**：
   - 所有外键字段必须建立索引
   - 常用查询条件建立联合索引
   - 避免过度索引，索引会增加写入开销
7. **字段命名**：使用下划线命名法（snake_case），如`user_name`、`last_message_time`
8. **注释**：所有表和字段都必须有清晰的注释

### 6.2 性能优化建议
1. **连接池**：使用数据库连接池（如Django的`CONN_MAX_AGE`），避免频繁创建和销毁连接
2. **查询优化**：
   - 避免`SELECT *`，只查询需要的字段
   - 使用`EXPLAIN`分析慢查询，优化索引
   - 分页查询使用`LIMIT`，避免全表扫描
3. **缓存策略**：
   - 热点数据（如用户信息）使用Redis缓存
   - MCP职位数据使用MySQL缓存，有效期7天
4. **定期维护**：
   - 定期清理过期的`mcp_jobs_cache`数据
   - 定期分析表，优化索引
   - 定期备份数据库

### 6.3 数据安全
1. **备份策略**：
   - 每日全量备份
   - 保留最近7天的备份
   - 备份文件存储在异地
2. **访问控制**：
   - 数据库用户只授予必要的权限
   - 禁止使用root用户连接应用
   - 限制数据库访问IP
3. **敏感数据**：
   - 密码后续可考虑使用bcrypt加密存储
   - 手机号可考虑脱敏展示（如138****8000）

### 6.4 监控与告警
1. **慢查询监控**：开启慢查询日志，定期分析
2. **连接数监控**：监控数据库连接数，避免连接池耗尽
3. **磁盘空间监控**：监控磁盘使用情况，及时扩容
4. **告警机制**：设置合理的告警阈值，及时发现和处理问题
