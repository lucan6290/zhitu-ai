# 01-项目立项.md
## 一、项目基本信息
- **项目名称**：职途AI - 岗位JD智能解析与学习路径规划助手
- **技术栈**：复用现有Face Pro框架（Django 6.0 + Vue 3 + LangGraph + MCP + ECharts 6.0 + 通义千问qwen3-max）

---

## 1.1 问题与市场分析
### 核心痛点（精准到具体场景）
解决**计算机相关专业大三/大四/研二学生**在找实习/校招时遇到的以下**具体、可感知**的问题：
1. **信息不对称**：不知道目标岗位**当前真实**的技术要求，网上的学习路线大多是1-2年前的，很多技术已经过时（比如还在教Flask而不是FastAPI）
2. **效率极低**：手动去BOSS直聘/猎聘翻30条JD，逐条复制粘贴整理技术栈，平均需要**1.5小时**，还容易遗漏高频考点
3. **优先级混乱**：不知道先学什么后学什么，把大量时间花在加分项上，必备技能反而没掌握（比如花1个月学Docker，却连MySQL索引都不会）
4. **不贴合地域**：一线城市和二三线城市的技术要求差异巨大，网上的通用路线不适合贵阳、成都等新一线城市的实习生岗位
5. **盲目学习**：学了很多和目标岗位无关的技术，投简历时发现岗位根本不要求，浪费了宝贵的求职准备时间

### 目标用户（精准画像）
| 维度 | 具体描述 |
|------|----------|
| **年龄** | 20-24岁 |
| **职业** | 计算机科学与技术、软件工程、人工智能、大数据等相关专业的大三、大四本科生，研二研究生 |
| **核心诉求** | 准备暑期实习、秋招提前批、春招补录，需要快速了解目标岗位的技术要求并制定学习计划 |
| **使用场景** | 1. 确定求职方向后，想知道"XX岗位需要学什么"<br>2. 投了200份简历没回音，想知道自己缺什么技能<br>3. 拿到面试通知，想快速突击该岗位的高频考点<br>4. 纠结两个岗位方向，想对比哪个技术要求更适合自己 |
| **用户规模** | 仅贵阳本地高校（贵大、贵师大、贵财等）计算机相关专业每年就有超过2000名实习生需求，全国市场规模百万级 |

### 现有方案及痛点
| 现有方案 | 用户使用方式 | 核心痛点 |
|----------|--------------|----------|
| **手动整理JD** | 自己去招聘网站翻30-50条JD，复制粘贴到Excel里统计技术栈 | 耗时1.5小时以上，容易出错，无法自动分级优先级 |
| **网上搜学习路线** | 知乎/B站搜索"2026年Python后端学习路线" | 内容过时、通用化严重，不针对具体城市和岗位，很多是培训机构的广告 |
| **问学长学姐** | 微信/QQ询问已经工作的学长学姐 | 经验可能过时（1-2年前的招聘情况），每个人的经历有限，无法代表整体市场 |
| **报培训班** | 花几千上万元报线下/线上培训班 | 费用高、周期长，内容同质化严重，不一定贴合当前企业的真实需求 |
| **其他AI工具** | 用ChatGPT直接问"Java实习生需要学什么" | 数据截止到2025年，没有实时招聘数据，回答非常笼统，没有针对性 |

### 我的优势
1. **技术优势**
   - 已有完整的**Face Pro前后端分离框架**，包含用户系统、AI智能体、SSE流式响应、ECharts可视化等核心能力，不需要从零搭建
   - 熟练掌握**LangGraph+MCP**架构，能快速集成第三方工具，MCP-Jobs直接解决了最复杂的爬虫和数据整合问题
   - 精通**大模型提示词工程**，能让大模型准确解析JD并生成符合要求的ECharts配置
   - 有**全栈开发能力**，能独立完成前后端所有开发工作

2. **资源优势**
   - **MCP-Jobs现成工具**：不需要自己写爬虫、处理反爬、维护多平台接口，节省至少2周的开发时间
   - **种子用户资源**：身边有大量正在找实习的同学，能快速获取50+种子用户进行测试和反馈
   - **通义千问API额度**：有充足的qwen3-max调用额度，MVP阶段不需要额外付费

3. **行业理解优势**
   - **自己就是目标用户**：正在找实习，深刻理解求职者的痛点和需求
   - **了解当前招聘市场**：已经投递了几十份简历，参加了多场面试，知道企业现在真正看重什么技术
   - **熟悉二三线城市招聘情况**：重点关注贵阳本地的实习岗位，能做出比通用工具更精准的分析

## 1.2 最小可行性产品（MVP）定义
### 第一步：列出所有能想到的功能
1. 多平台JD聚合获取（BOSS直聘、猎聘、智联、前程无忧）
2. JD结构化解析（技术栈、岗位职责、软技能、经验要求、学历要求）
3. 技术栈频率统计与三级优先级分级（必备/重要/加分）
4. 技术栈领域分类（后端、数据库、前端、工具、云服务等）
5. ECharts可视化展示（技术栈频率柱状图、领域分布饼图、经验要求柱状图）
6. 基于用户专业背景的个性化学习路径生成（分1周/2周/1个月）
7. 用户技能自评与差距分析（雷达图）
8. 高频面试题生成
9. 匹配岗位推荐
10. 分析报告导出为PDF
11. 用户登录注册
12. 聊天历史记录保存
13. 多会话管理
14. 分享功能
15. 付费功能

### 第二步：用"没有这个功能，核心流程就走不通"的标准筛选
**核心流程**：用户输入「专业+目标岗位+意向城市」→ 获取数据 → 分析技术栈 → 生成可视化 → 输出学习计划

### 第三步：最终MVP核心功能（仅保留4个）
| 序号 | 核心功能 | 为什么必须有 |
|------|----------|--------------|
| 1 | **多平台JD聚合获取**（基于MCP-Jobs） | 没有数据就无法进行任何分析，是整个项目的基础 |
| 2 | **技术栈智能解析与优先级分级** | 项目的核心价值，解决用户"不知道需要学什么、先学什么"的核心痛点 |
| 3 | **ECharts可视化展示** | 区别于其他AI工具的核心亮点，让分析结果更直观、更有说服力 |
| 4 | **基础学习路径生成** | 给用户提供可执行的行动方案，完成从"知道问题"到"解决问题"的闭环 |

### 所有其他功能全部放到后续迭代
- 迭代1（MVP上线后1周）：用户技能自评与差距分析、高频面试题生成
- 迭代2（上线后2周）：聊天历史记录、分析报告导出
- 迭代3（上线后1个月）：用户系统、岗位推荐、付费功能

## 1.3 可行性验证
### 技术可行性验证（POC计划，1天完成）
针对项目中**最不确定的3个技术点**进行概念验证，确保MVP能顺利开发：

#### POC 1：MCP-Jobs服务集成验证
- **验证目标**：确认能正常调用MCP-Jobs工具，获取多平台的结构化JD数据
- **验证步骤**：
  1. 申请MCP-Jobs免费API密钥
  2. 在现有FastMCP客户端中配置MCP-Jobs服务
  3. 调用`search_jobs(keyword="Python后端实习生", city="贵阳", limit=10)`工具
  4. 检查返回数据是否包含岗位名称、公司、薪资、岗位职责、任职要求等字段
- **成功标准**：能正常获取至少10条来自不同平台的结构化JD数据，无报错

#### POC 2：大模型JD解析与技术栈统计验证
- **验证目标**：确认通义千问qwen3-max能准确解析JD文本，统计技术栈频率并分级
- **验证步骤**：
  1. 准备10条真实的Python后端实习生JD
  2. 编写提示词，让大模型提取技术栈、统计频率、按必备/重要/加分分级
  3. 人工检查解析结果的准确性
- **成功标准**：技术栈提取准确率≥90%，频率统计误差≤10%，分级逻辑正确

#### POC 3：大模型生成ECharts配置验证
- **验证目标**：确认大模型能生成标准的ECharts 6.0配置JSON，前端能正常渲染
- **验证步骤**：
  1. 给大模型提供技术栈频率统计数据
  2. 编写提示词，要求生成柱状图的ECharts配置
  3. 将生成的配置复制到ECharts官方示例中测试渲染
- **成功标准**：生成的配置无语法错误，能在ECharts中正常渲染，图表展示正确

#### POC 4：完整流程验证
- **验证目标**：确认从用户输入到返回结果的完整流程能正常运行
- **验证步骤**：
  1. 用户输入"我是计算机专业，想投贵阳的Python后端实习生"
  2. Agent自动调用MCP-Jobs获取数据
  3. 大模型解析数据、生成分析结果和ECharts配置
  4. 前端渲染文字和图表
- **成功标准**：整个流程在30秒内完成，无报错，结果准确

## 1.4 成功与失败边界
### 成功标准（MVP阶段）
1. **功能完成度**：4个核心功能全部实现，核心流程成功率≥95%（即用户输入后能正常返回结果，无报错）
3. **反馈指标**：收集到至少20条有效用户反馈，明确后续迭代方向


### 失败边界（满足任意一条即放弃项目）
1. **技术失败**：MCP-Jobs服务不可用，且在1周内找不到替代的JD数据获取方案，导致核心功能无法实现
2. **时间失败**：开发时间超过3周，且MVP核心功能仍未完成
3. **需求失败**：MVP上线后1周内，用户量不足10人，或用户满意度<3分，且没有明确的改进方向
